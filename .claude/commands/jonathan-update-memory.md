你的工作原则：
- **不遗漏，不虚构** — 只记录真实发生的变更，既不漏掉关键改动，也不为了充实档案而编造内容
- **先确认，再动笔** — 每一步都与用户对齐，不擅自决定记录什么、怎么记录
- **为下一个人写** — 你写的每一行都是给一个对 Jonathan 一无所知的新 session 看的，确保他能看懂、能用上

更新 Jonathan 记忆文件夹。回顾本次对话中对 Jonathan 做出的操作和发现，将有价值的信息持久化到记忆文件中。

## 前置检查

如果本次会话没有对 Jonathan 做任何实际操作或产生新发现（例如只是闲聊、只是查看状态），直接告知用户"本次会话无需更新记忆文件"，然后**仅执行步骤 2a（一致性校验）和步骤 4（增长控制）** 确保系统健康，跳过其余步骤。执行完后向用户简要汇报一致性校验和增长控制的结果。

## 工作流程

### 1. 盘点本次会话的变更

回顾对话内容，**逐条列出**所有值得记录的变更。不限类别，任何对 Jonathan 的运行、配置、环境、能力认知有影响的变更都应记录。

输出格式（必须）：
```
本次会话变更清单：
1. 具体变更描述
2. 具体变更描述
...
```

**列出清单后，询问用户确认**："以上是我整理的变更清单，是否有遗漏或需要补充？"
- 用户确认 → 继续步骤 2
- 用户指出遗漏或错误 → 修正清单，重新输出，再次确认。循环直到用户满意。

### 2. 判断更新方式

#### 2a. 一致性校验

先检查 README 与实际文件是否一致。

在项目根目录下运行：
```bash
ls *_*.md 2>/dev/null
ls archive/*_*.md 2>/dev/null
```
（如果当前目录不在项目根，先 cd 到项目根目录。项目根 = 包含 README.md 和 `YYYY-MM-DD_*.md` 文件的目录。）

对比 README 表格。如发现不一致（文件缺失、未注册、同主题多文件冲突），纳入本次修复范围。

#### 2b. 变更归属与方式判断

对步骤 1 确认后的每项变更：
1. 读取所有当前层文件的 YAML frontmatter（`tags`、`status`），作为归属判断的**参考**
2. 结合 tags 和文件实际内容，确定每项变更属于哪个文件（tags 是辅助线索，不是唯一依据 — 变更的语义与文件主题的相关性才是决定因素）
3. 如果变更不属于任何现有文件 → **新建文件**
4. 如果属于某个现有文件，读取该文件内容，判断更新方式：
   - 只需修改少量具体数值/状态，文件整体结构不变 → **原地编辑**
   - 文件的核心结论或整体结构需要推翻重写 → **归档 + 新建**
   - 介于两者之间（不确定）→ 在更新计划中标注，向用户说明情况，让用户决定

输出格式（必须）：
```
更新计划：
- 变更 1 → 原地编辑 `filename.md`（改 XX 为 YY）
- 变更 2 → 归档 `old.md` + 新建 `new.md`（原因：核心结论需推翻）
- 变更 3 → 新建 `YYYY-MM-DD_topic.md`（新主题）
- 变更 4 → 需用户决定：原地编辑还是归档重写？（原因说明）
```

如果有需要用户决定的项，等用户回复后，将决定更新到计划中再执行。

**原地编辑**：
- 修正具体数值、状态等
- 在文件 YAML frontmatter 中更新 `last_updated: YYYY-MM-DD`
- 编辑后必须重新读取该文件，确认改动正确

**归档 + 新建**：
- 将旧文件移入 `archive/` 目录（自动创建）：
  ```bash
  mkdir -p archive && mv YYYY-MM-DD_topic.md archive/
  ```
- 创建新日期的同主题文件，保持相同的 `_topic` 后缀
- 新文件 frontmatter 中添加 `supersedes: archive/旧文件名.md`

**新建文件**：
- 按 `YYYY-MM-DD_topic-name.md` 格式创建
- 包含标准 YAML frontmatter（见步骤 5）
- 在步骤 3 维护 README 时，为新文件分配 Priority 编号和"何时加载"描述，插入到 Reading Order 表格的合适位置

### 3. 维护元文件

**必须先读再判断** — 不要凭猜测决定"不需要改"，每个元文件都要实际读取内容后再做决定。

#### README.md（索引层）
1. 读取 README.md
2. 逐项对照步骤 1 的变更清单：
   - Reading Order 表格是否需要增删文件？
   - 表格与实际文件是否一致？（步骤 2a 的校验结果）
   - TL;DR 中的**每一条**逐条检查是否仍然准确
3. 如有变化则编辑；如无变化，明确说明"已读取 README.md，逐条比对结果："并列出每条的比对结论

#### .claude/CLAUDE.md（指令层）
1. 读取 .claude/CLAUDE.md
2. 逐项对照步骤 1 的变更清单，检查 CLAUDE.md 中的**每个章节**是否需要更新
3. 如有变化则编辑；如无变化，明确说明"已读取 CLAUDE.md，逐节比对结果："并列出每节的比对结论

不要为了改而改，但必须展示"读过 → 比对过 → 决定不改"的完整过程。

#### .claude/rules/（规则层）和 CLAUDE.local.md（本地配置层）

这些文件每次 session 自动加载到 system prompt，变更影响所有未来对话。

1. 判断本次变更是否涉及以下任一类别：
   - 操作流程或命令变化 → 检查 `.claude/rules/jonathan-ops.md`
   - 记忆格式或容量规范变化 → 检查 `.claude/rules/memory-format.md`
   - 敏感文件处理规则变化 → 检查 `.claude/rules/sensitive.md`
   - 服务器连接信息变化（IP、端口、SSH 配置） → 检查 `CLAUDE.local.md`
2. 如涉及，读取对应文件，判断是否需要同步更新
3. 如不涉及，明确说明"本次变更不涉及规则层和本地配置层"，跳过

### 4. 文件增长控制

**每次执行必须检查**：

在项目根目录下运行：
```bash
echo "当前层文件数:"; ls *_*.md | wc -l
echo "各文件行数:"; wc -l *_*.md
echo "archive/ 中同主题计数:"; ls archive/ 2>/dev/null | sed 's/^[0-9-]*_//' | sort | uniq -c | sort -rn
```

根据输出判断：
- 当前层文件 > **8 个** → 必须合并相关主题
- 单个文件 > **150 行** → 必须拆分或精简
- `archive/` 中同主题文件 > **3 个** → 必须编写演变摘要

如果所有指标正常，输出："增长检查通过：N 个文件，最大 M 行，无需压缩。"

### 5. YAML Frontmatter 规范

每个文件必须包含：
```yaml
---
title: 文件标题
date: YYYY-MM-DD          # 创建日期
last_updated: YYYY-MM-DD  # 最后修改日期（原地编辑时更新）
tags: [tag1, tag2]        # 用于渐进式加载和变更归属判断（辅助线索）
depends_on: []            # 阅读前需要先读的文件
status: current           # current | superseded | archived
supersedes: ""            # 如果替代了旧文件，填旧文件路径
---
```

### 6. 完成后

向用户汇报（必须包含以下所有项）：
- 变更清单回顾（步骤 1 确认的 N 项）
- 每项变更的处理结果：修改了什么 / 归档了什么 / 新建了什么
- 一致性校验结果：README 与实际文件是否同步
- 元文件检查结果：README 和 CLAUDE.md 是否有修改及原因
- 增长检查结果：文件数、最大行数、是否触发压缩
- **用户确认**："以上更新是否正确？有需要调整的地方吗？"
